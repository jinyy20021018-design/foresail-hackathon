How ForeSail Works — System Workflow

This document explains the end-to-end workflow of ForeSail: how a trade case moves from document upload through field confirmation, agent monitoring, risk mapping, and action follow-up. It covers both the user journey and the backend pipeline.

MVP 3.0 White-paper style workflow 中文用户指南 · English Guide

1. What ForeSail Does

ForeSail is a case-aware trade shipment monitor. Each shipment is one Case. The system reads trade documents, builds a watch profile, ingests external events, scores relevance, maps exposures, and recommends actions — while keeping core business decisions deterministic.

Design principle: Scoring, classification, status transitions, exposure mapping, and action deduplication are deterministic rules. LLM is optional and limited to document extraction, event text parsing, and user-facing summaries — never for final risk decisions.

3. User Workflow (7 Phases)

This is what a trade ops user experiences in the UI, tracked by the Workflow Stepper.

1

Upload Documents

Upload Contract/PO (required), Booking Confirmation, LC, and optional supporting files to a Case.

2

Review Extracted Fields

System extracts vessel, ports, dates, payment terms, etc. User approves, edits, or rejects each field with evidence.

3

Resolve Conflicts

If Contract, Booking, and LC disagree (e.g. dates), high-severity conflicts must be resolved before proceeding.

4

Confirm Case Facts

User clicks Confirm Fields. These become the authoritative facts for monitoring.

5

Run Agent Monitoring

User triggers Run Agent Monitoring Cycle. Agent ingests events, scores relevance, maps risks.

6

Review Outputs

Review obligations, information gaps, recommended actions, action drafts, and treatment plans.

7

Continue Monitoring

When status is ACTION_REQUIRED, user acknowledges and clicks Continue Monitoring → MONITORING.

Gate rule: Agent run is blocked if (a) fields are not confirmed, or (b) unresolved high-severity conflicts exist.

4. Document Ingestion Pipeline

When documents are uploaded and extracted, the backend runs this pipeline per file.

  1. Upload & store file File saved under backend/app/uploads/{case_id}/. Document record created with type (CONTRACT_PO, BOOKING_CONFIRMATION, LETTER_OF_CREDIT, etc.).
  2. Detect format TXT, DOCX, or PDF. PDFs are checked for text-based vs scanned (pdf_detection_service).
  3. Extract raw text Text extraction runs first. Scanned PDFs may return NEEDS_VISION unless vision mode is enabled.
  4. LLM extraction (optional) If USE_LLM_EXTRACTION=true, OpenAI extracts structured JSON fields with evidence snippets (document_service._llm_extract_fields). Otherwise regex/deterministic parsing is used.
  5. Validate schema Extracted fields validated against allowed field names and document-type expectations.
  6. Detect conflicts Cross-document comparison finds mismatches (e.g. LC latest shipment date vs contract). Severity: High / Medium / Low.
  7. Human review User approves/edits/rejects fields in the Documents & Evidence tab.
  8. Confirm facts Confirmed fields become the case's operational truth and feed the watch profile builder.
Deterministic parsing & validation Optional LLM extraction

6. Agent Monitoring Pipeline (Backend Detail)

MonitoringAgent.run_monitoring_cycle() orchestrates ~20+ trace steps. Below is the logical flow.

Phase A — Preconditions

  1. Load Case — Read case snapshot and current status.
  2. Check field conflicts — If high-severity OPEN conflicts exist, agent run is BLOCKED.
  3. Load confirmed facts — Use confirmed fields; fall back to case snapshot if none.
  4. Build watch profile — Vessel, ports, regions, deadlines to monitor.

Phase B — Event ingestion

  1. Build search queries — From watch profile (search_query_builder).
  2. Fetch events — Via connectors based on EVENT_SOURCE_MODE:
    • MOCK — Built-in demo events (default)
    • REAL — GDELT news + Open-Meteo weather
    • HYBRID — Both mock and real
  3. Normalize events — Convert to unified external event schema.
  4. Deduplicate — Remove duplicates by dedup key and source confidence.

Phase C — Deterministic analysis (core decisions)

  1. Classify event relevance (relevance_engine)
    Each event scored and labeled:
    • Relevant — score ≥ 70
    • Watch — score 40–69
    • Irrelevant — score < 40
    Factors: vessel match, port match, route region, shipment window, severity, weather cap.
  2. Map exposures (risk_mapper) — Aggregate Relevant/Watch events into exposure categories (Shipping, LC Deadline, Port Operation, Payment Timeline).
  3. Map obligations & deadlines (obligation_service) — LC presentation, shipment dates, payment timeline assessments.
  4. Detect information gaps (information_gap_service) — Missing confirmations that block decisions.
  5. Generate actions (action_board_service) — Deduplicated recommended actions from triggered exposures.
  6. Generate action drafts (action_draft_service) — Draft messages for review (nothing sent externally).
  7. Update case status (status_machine) — Transition based on monitoring results.

Phase D — Treatment & summary

  1. Generate treatment plans — Structured options for handling identified risks.
  2. Generate approval package — Draft approval summary for recommended plan.
  3. Save agent run history — Full trace persisted for audit.
  4. Generate agent summary — Deterministic or optional LLM user-facing summary (agent_summary_service).
Agent orchestration Deterministic business rules Optional LLM summary only

10. Where LLM Is (and Is Not) Used

Status transitionsState machineNo—

Agent run summaryDeterministic or OpenAIOptionalUSE_LLM_SUMMARY

Why this matters: ForeSail separates orchestration (Agent layer) from decision logic (deterministic services). The Agent coordinates tools; it does not invent scores or status changes via LLM hallucination.
